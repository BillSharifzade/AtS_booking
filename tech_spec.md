Общая концепция системы бронирования помещений
1. Общая концепция системы
Система представляет собой платформу автоматизации бронирования помещений и организации мероприятий через Telegram-бота и административную панель.
1.1 Пользователь создает заявку на бронирование помещения, указывает параметры мероприятия и получает подтверждение либо альтернативные варианты времени.
1.2 Система автоматически проверяет доступность помещений, наличие ресурсов (например, кофе-брейка), лимиты участников и корректность временных интервалов.
        1.3 Все заявки и статусы хранятся в единой базе данных (Excel/БД).
1.4 Система поддерживает автоматические уведомления администраторам, заказчикам и ответственным подразделениям.
        1.5 После завершения мероприятия заявка архивируется, а пользователю может быть отправлена feedback-форма.
2. Основные роли пользователей
2.1. Заказчик:
• Создание заявки на бронирование
• Просмотр доступных временных слотов
• Получение уведомлений и подтверждений
• Выбор альтернативного времени при занятости помещения
• Получение напоминаний о мероприятии
2.2. Администратор
• Проверка и обработка заявок
• Управление слотами и помещениями
• Контроль кофе-брейков и дополнительных услуг
• Мониторинг всех бронирований
• Отправка уведомлений в Telegram-группы
2.3. Руководители/Ответственные подразделения
• Получение информации о предстоящих мероприятиях
• Контроль загруженности помещений
• Мониторинг крупных мероприятий
3. Функциональные требования
3.1. МОДУЛЬ A - Создание заявки
Создание запроса на бронирование
• Компания
• ФИО контактного лица
• Телефон
• Тип помещения
• Дата
• Время начала/окончания
• Тип мероприятия
• Название мероприятия
• Описание мероприятия
• Количество участников
• Наличие кофе-брейка
Проверка роли пользователя
• Определение роли (Заказчик/Администратор)
• Ограничение функционала в зависимости от роли
Проверка срочного бронирования
• Если до мероприятия менее 2 дней - требуется связь с Администратором
• Возможность отказа при нарушении правил только под контролем Администратора

3.2 МОДУЛЬ B - Проверка и обработка заявки
Автоматическая проверка данных
• Проверка обязательных полей
• Проверка корректности даты и времени
• Проверка лимита участников
• Проверка необходимости кофе-брейка
Проверка доступности слота
• Проверка свободного времени в Базе Данных (БД)
• Проверка пересечений мероприятий
• Проверка доступности помещения
Предложение альтернатив
• Если слот занят - система предлагает доступные варианты
• Заказчик может выбрать новый вариант времени

3.3 МОДУЛЬ C - База данных и хранение
Хранение заявок
• Все заявки сохраняются в Excel/БД
• Хранение статусов и истории изменений
Управление статусами
• Новая заявка
• На обработке
• Подтверждено
• Отклонено
• Завершено
• Архивировано

3.4 МОДУЛЬ D - Подтверждение и уведомления
Подтверждение бронирования
• После проверки Администратор подтверждает заявку
• Создается официальная бронь
Уведомления
• Уведомление Администратора
• Уведомление Заказчика
• Уведомление в Telegram-Группу
Автоматические напоминания
• Напоминание за 1 день
• Напоминание за 1 час

3.5 МОДУЛЬ E - Управление кофе-брейком и зонами
Работа с зонами
• Зона A
• Зона Б
• Зона В
Логика распределения
• Приоритет у того, кто забронировал раньше
• Администратор может скорректировать распределение по загруженности кофе-брейка
Подготовка кофе-брейка
• Учет необходимости кофе-брейка
• Подготовка кофе-брейка в помещении при наличии гостей

3.7 МОДУЛЬ F - Завершение мероприятия
Завершение бронирования
• Изменение статуса
• Фиксация результата
• Архивирование заявки
Обратная связь
• Отправка feedback-формы заказчику
• Сбор обратной связи по мероприятию
